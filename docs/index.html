01 The paper and the method

A serious manuscript review is more than a list of reactions. Claims move between drafts, page numbers drift, related concerns multiply, and a final report can become detached from the evidence that motivated it. Katz keeps a review as a repository-scoped ledger tied to a Git commit and a preserved copy of the manuscript.

This tutorial reviews a real, short software paper: Roni W. Kobrosly’s causal-curve: A Python Causal Inference Package to Estimate Causal Dose-Response Curves, published in the Journal of Open Source Software in 2020. The paper is useful here because its Markdown source is public, the published article is licensed CC BY 4.0, and its completed public peer review gives us an external record against which to compare our process.

This is a workflow demonstration, not a new assessment of the author or published article. We review the source to learn Katz. JOSS’s accepted version and public review remain the authoritative publication record.

Every registered manuscript is associated with a repository commit. Findings use byte ranges and section IDs, while spotters, issue states, investigation notes, and suggestions make the review comparable across revisions. The local .katz/ directory preserves source copies, decisions, evaluations, and reports.

The working sequence is to register a version, map its sections, find and investigate issues, evaluate and validate the ledger, and generate a report.

Katz does not review the paper for you. Spotters define useful questions and the ledger preserves conclusions, but a reviewer or agent must still read the manuscript, assess evidence, and file concrete findings.

02 Set up the worked example

Install Katz and EDSL, then clone the paper’s repository. We create a branch only to keep the tutorial’s Katz state separate from the upstream project. Katz deliberately places .katz/ at the repository root.

Install Katz and EDSL

python -m pip install \
  git+https://github.com/expectedparrot/katz.git

python -m pip uses the selected Python interpreter’s installer. Katz installs the current EDSL Git dependency that provides jobs.ep, results.ep, and the ep CLI used later.

Configure Expected Parrot

The agent-run review step requires an Expected Parrot account and a valid API key. Create a file named .env in the repository root containing:

EXPECTED_PARROT_API_KEY=your-valid-key-here

EDSL reads this key when ep run executes the jobs.ep object later in the tutorial. Keep .env local: add it to .gitignore and never commit or paste the real key into a job, result, report, or issue record. If your selected model requires a separate provider key, add that provider’s environment variable to the same local file.

Clone the paper repository

git clone https://github.com/ronikobrosly/causal-curve.git

git clone creates a local repository with the paper source and its version history.

cd causal-curve

cd makes that repository the working directory for every subsequent Katz command.

git switch -c katz-tutorial

switch -c creates and checks out a new branch named katz-tutorial, keeping tutorial artifacts separate from the upstream branch.

Initialize Katz state

katz init

katz init creates the local directory structure but no tracked file yet. The existing upstream commit is therefore the version identity used at registration; commit the generated review state afterward if you want it in the paper repository.

The upstream source is paper/paper.md. Before continuing, open it and the published paper page. The source repository may evolve after publication, so your Git commit may differ from the accepted 2020 version.

Create a ventilated derivative before registration. The command leaves the upstream paper untouched and returns one JSON envelope:

--no-index compares two paths directly, even before the generated file is tracked.

-m supplies the commit message. This commit becomes the Katz version identity.

Git is part of the data model. Commit before registration. A meaningful commit gives the review a stable version identity and makes this tutorial’s result reproducible.

03 Starting from a PDF with paper2md

Our worked example begins with the author’s Markdown source, which is the best available input. When only a PDF is available, use paper2md to create a reviewable bundle before invoking Katz.

python -m pip install \
  "paper2md @ git+https://github.com/expectedparrot/paper2md.git"

This direct-reference requirement installs paper2md from its Expected Parrot Git repository.

paper2md paper.pdf \
  --backend marker \
  --output paper_md
Argument or optionMeaning
paper.pdfSource PDF to convert.
--backend markerUses Marker for layout-aware Markdown, equations, captions, and tables.
--output paper_mdWrites paper.md and extracted images into this directory.

The output directory contains paper_md/paper.md and normalized PNG assets such as img_1.png. Markdown image references are rewritten to those filenames. Images extracted from the PDF but not referenced by the converted text are appended under an Extracted Figures heading so they are not silently lost.

For figures, paper2md extracts embedded images, writes PNG files, and retains Markdown references. Captions depend on the conversion backend and source PDF. For tables, the Marker backend attempts to reconstruct them in Markdown. They remain blocks inside paper.md rather than separate table records or image assets.

Inspect the conversion

PDF conversion is lossy. Before registration, compare the bundle with the PDF and check the parts most likely to affect a review:

find paper_md -maxdepth 1 -type f -print

-maxdepth 1 limits inspection to the bundle root; -type f selects files; -print lists each Markdown or image asset.

open paper.pdf

On macOS, open launches the source PDF in its default application.

open paper_md/paper.md

Open the converted Markdown separately and compare its reading order, equations, figures, and tables with the PDF.

CheckTypical failure
Reading orderMulti-column text is interleaved or moved beneath a figure.
Figures and captionsAn image is duplicated, omitted, cropped, or paired with the wrong caption.
TablesRows or columns shift, headers flatten, and significance markers detach.
EquationsSymbols, subscripts, alignment, or equation numbers are damaged.
ReferencesCitations and cross-references lose labels or become conversion noise.

Ventilate and commit the bundle

paper2md often produces useful line breaks, but Katz should verify and normalize the prose itself. Write a separate ventilated file inside the same directory so its relative PNG references continue to resolve:

katz ventilate paper_md/paper.md \
  --output-path paper_md/paper_ventilated.md

The positional argument is paper2md’s Markdown output; --output-path keeps the derivative beside its referenced images.

git diff --no-index \
  paper_md/paper.md \
  paper_md/paper_ventilated.md

Inspect every prose-only change before accepting the derivative.

git add paper.pdf paper_md/

Stage the source PDF and complete converted bundle together so provenance and relative image references remain intact.

git commit -m "Add paper2md review bundle"

Commit the prepared bundle before registration so Katz can key it to an immutable Git SHA.

Register the committed derivative and record the conversion provenance:

katz init

Initialize repository-scoped .katz/ state.

katz paper register \
  --canonical paper_md/paper_ventilated.md \
  --source-format pdf \
  --source-method paper2md-marker \
  --source-root paper.pdf
OptionMeaning
--canonicalThe prepared Markdown file Katz copies and anchors.
--source-format pdfRecords the original source medium, not the canonical file extension.
--source-method paper2md-markerRecords the conversion method as provenance.
--source-root paper.pdfPoints back to the original PDF in the repository.

Current Katz boundary. Registration copies sibling PNG/JPEG/GIF/SVG/WebP assets into the version bundle, but it does not parse Markdown image references into figure records. Tables remain manuscript text and have no first-class records. Preserve and inspect the paper2md bundle even though katz paper status currently reports "figures": 0.

Prefer source when possible. If author-provided Markdown or LaTeX exists, use it instead of reverse-engineering the PDF. paper2md is the fallback for published papers whose structured source is unavailable.

04 Register a manuscript

Registration copies the committed paper/paper_ventilated.md into versioned Katz state, records its checksum, and indexes its prose into addressable sentences. The current causal-curve repository commit identifies the exact review version.

katz paper register \

  --source-format markdown \
  --source-method ventilated-published-joss-source

--canonicalThe authoritative file to preserve and review.
--source-formatThe source representation, such as Markdown or LaTeX.
--source-methodHow the artifact arose: manuscript, revision, or generated report.

The envelope is the default API. Every Katz command writes one JSON document to stdout. Agents first branch on ok, use data only when it is true, and use error.code, error.message, and error.details when it is false. command records the exact invocation arguments.

Why does status say zero figures? Registration currently copies sibling image assets but does not create figure records from Markdown references. Tables remain embedded manuscript text and have no first-class registry. The zero is therefore an implementation limitation, not a claim that the paper contains no visual or tabular material.

Register one canonical artifact per version. If appendices or supplementary files matter, mention them in issue evidence or artifacts rather than silently treating multiple files as one source.

06 Add and enable spotters

A spotter is a Markdown instruction file: YAML frontmatter declares whether it applies to each section or the whole manuscript, the main body tells the reviewer or model what to look for, and an optional ## Investigation block explains how to confirm or reject a candidate finding.

We are going to apply enabled spotters to the registered manuscript. For a section-scoped spotter, Katz pairs the instructions with each mapped section; for a holistic spotter, it pairs them with the complete paper. The next chapter sends those combinations to reviewing models and turns returned candidates into manuscript-anchored issues.

What a spotter looks like

A spotter is ordinary Markdown that an agent can read and edit. This is the complete built-in causal_language.md definition we will use:

---
scope: section
---
# Causal Language

Check whether causal language is appropriate for the research design.

Pay special attention to:
- Words like "causes", "leads to", "increases", "reduces", "impact of" used to describe correlational or observational findings
- Whether the identifying assumptions are explicitly stated (e.g., conditional independence, exclusion restriction, parallel trends)
- Whether the author specifies what would need to be true for the estimates to have a causal interpretation
- Interaction terms interpreted as causal differences across subgroups without justification
- Null findings interpreted as "no effect" rather than "we cannot reject zero" — conflating absence of evidence with evidence of absence
- Conditioning on post-treatment variables (bad controls)

## Investigation

- Check the research design section for the stated identification strategy
- Determine whether the causal language matches the design (experiment, IV, RD, diff-in-diff, or observational)
- If observational, check whether the author uses hedged language ("is associated with") or causal language ("causes")
- **Confirm** if causal language is used without a credible identification strategy
- **Reject** if the design supports the causal claim or the language is appropriately hedged
PartHow Katz uses it
scope: sectionApplies this spotter separately to every registered manuscript section. Use holistic when it needs the entire manuscript.
# Causal LanguageProvides the human-readable title shown by catalog and inspection commands.
Main instructionsTell the reviewing agent what evidence and failure patterns to look for.
## InvestigationTells the agent how to verify a candidate and when to confirm or reject it.
FilenameSupplies the stable identifier causal_language used by enable, run, and issue commands.

Spotters are review procedures, not keyword lists. A useful definition states the standard, identifies relevant evidence, and explains how to disconfirm a tempting false positive.

How Katz works with spotters

  1. Catalog: keep reusable Markdown definitions in .katz/spotters/.
  2. Enable: copy selected definitions into the active manuscript version, freezing the exact review instructions.
  3. Apply: combine enabled spotters with registered sections or the complete manuscript and run the resulting review jobs.
  4. Record: store each candidate as an issue with its spotter name, manuscript byte range, and model context.
  5. Investigate: confirm or reject the candidate using the spotter’s investigation procedure.

Populate the repository catalog

katz spotter init-catalog

Copies the built-in default collection into .katz/spotters/, skipping definitions already present. Use --preset NAME to select another installed collection.

Inspect available definitions

katz spotter catalog

Lists spotters available in the repository catalog. Add --scope section or --scope holistic to filter by execution scope.

Enable two lenses for this version

katz spotter enable causal_language

The positional name selects .katz/spotters/causal_language.md. Katz copies it into the active version, freezing the instructions used for this review. Optional --commit SHA targets another registered version.

katz spotter enable overclaiming

Enables the same way, but looks for conclusions whose strength exceeds the available evidence.

Verify the enabled set

katz spotter list

Lists definitions enabled for the active version. --scope filters the list and --commit selects another version.

Add a new spotter for this paper

The causal-curve article is a software paper, so we add a lens that asks whether statements about implemented functionality are supported by evidence or a reproducible pointer. Because the claim may appear in one section while support appears elsewhere, this is a holistic spotter.

katz spotter add \
  --name software_claim_support \
  --scope holistic \
  --description "Identify concrete claims about implemented estimators, supported outcomes, performance, or validation. Flag a claim only when the paper provides neither evidence nor a specific reproducible pointer to documentation, tests, examples, or cited literature." \
  --investigation "Check the complete paper, linked documentation, examples, tests, and the public JOSS review. Confirm only if the support is materially absent; reject if another section or cited artifact supplies it."
OptionMeaning
--nameHuman-readable identifier; Katz slugifies it for the filename and issue field.
--scope holisticRuns once on the complete manuscript per model rather than once per section.
--descriptionRequired instructions defining what evidence should trigger the spotter.
--investigationOptional guidance for confirming or rejecting generated candidates.
katz spotter catalog-show software_claim_support

Shows the catalog copy, including parsed scope, title, description, and investigation guidance.

katz spotter show software_claim_support

Shows the enabled version copy—the exact instructions that EDSL will use.

spotter add slugifies the name, writes .katz/spotters/software_claim_support.md, and auto-enables the same file for the active version. If there is no registered active version yet, it adds only the catalog copy; run katz spotter enable software_claim_support after registration.

Choose scope deliberately. A section spotter produces one scenario per section. A holistic spotter sees the complete manuscript once per model. Use holistic scope when evidence must be reconciled across sections.

A focused set usually produces a better review than enabling every category. Software papers often need spotters for statement of need, state of the field, claim support, examples, tests, and documentation. Theory reviews need logical gaps, construct clarity, positioning, mechanisms, and boundary conditions. Editorial reviews emphasize contribution, narrative flow, tables and figures, limitations, and claim scope. Generated-report QA emphasizes evidence–claim alignment, missing artifacts, interpretation, and unsupported recommendations.

07 Build, run, and ingest EDSL jobs

Katz prepares the review but does not run models. It combines the active manuscript version, mapped sections, and enabled spotters into a standard EDSL Jobs object. The object is saved as jobs.ep, so it can be inspected, costed, run, shared, or archived with the normal ep CLI.

The scenario count is sections × section spotters + holistic spotters.

Build the Jobs object

katz spotter jobs --output jobs.ep
OptionMeaning
--output jobs.epWrites a real EDSL Jobs package. Katz refuses to overwrite an existing package.
--section IDOptional: limits section-scoped spotters to one mapped section. Holistic spotters still use the complete manuscript.
--spotters a,bOptional: includes only these comma-separated enabled spotters.
--commit SHAOptional: builds from another registered Katz version instead of the active one.

For this paper, five sections crossed with the two enabled section spotters produce ten section scenarios; software_claim_support adds one holistic scenario:

{
  "ok": true,
  "command": ["spotter", "jobs", "--output", "jobs.ep"],
  "data": {
    "object_type": "Jobs",
    "output": "jobs.ep",
    "commit": "c658feac957d3052e30da8e0365e4d982d091d9a",
    "question": "spotter_result",
    "spotters": [
      "causal_language",
      "overclaiming",
      "software_claim_support"
    ],
    "scenario_count": 11,
    "section_scenarios": 10,
    "holistic_scenarios": 1,
    "saved": {
      "status": "ok",

      "commit": "$JOBS_PACKAGE_COMMIT"
    },
    "next": "ep run jobs.ep --model <model-name> --output results.ep"

}

The survey inside the package uses EDSL’s structured dictionary question type. Every result has found, title, quoted_text, and description fields, avoiding an ad hoc JSON parser.

Inspect and estimate before running

ep inspect jobs.ep

Loads the package without model calls and reports that it contains an EDSL Jobs object, including its survey, scenario, and model counts.

ep jobs cost jobs.ep

Estimates the cost of the packaged job. Add --iterations N to estimate repeated runs.

Run with the EDSL CLI

ep run jobs.ep \
  --model gpt-4.1-mini \
  --output results.ep
Argument or optionMeaning
jobs.epThe portable Jobs package created by Katz.
--modelSelects the model at run time; the Katz package does not hard-code one.
--output results.epPreserves the complete EDSL Results object, including answers, scenarios, model metadata, and run provenance.

Use --local to disable remote inference. For an asynchronous remote run, use --background, then retrieve the completed object with ep jobs results JOB_UUID --output results.ep.

Import candidate issues into Katz

katz spotter ingest results.ep

The positional path must contain an EDSL Results object. Katz verifies the embedded manuscript commit and enabled spotter, locates the exact quoted text within the original scenario range, and files valid findings as draft issues. --state STATE changes the initial state; --commit SHA selects a registered version.

Because model findings and UUIDs vary between runs, the tutorial uses a deterministic completed-run fixture with three positive findings. Ingestion returns:

{
  "ok": true,
  "command": ["spotter", "ingest", "results.ep"],
  "data": {
    "results": "results.ep",
    "commit": "c658feac957d3052e30da8e0365e4d982d091d9a",
    "result_count": 11,
    "issues_found": 3,
    "issues_filed": 3,
    "skipped": 0,
    "issue_ids": [
      "ISSUE_CAUSAL_LANGUAGE",
      "ISSUE_UNBIASED_EFFECT",
      "ISSUE_CLAIM_SUPPORT"
    ]
  }
}

The fixture uses readable issue labels because generated UUIDs vary from run to run. The counts and meanings are the fields an agent should inspect: eleven EDSL results, three positive findings, three filed issues, and no unlocatable quotations.

Answers with found: false are retained in results.ep but do not become issues. Answers whose quotation cannot be found are skipped rather than assigned a guessed anchor. Re-ingesting the same results is idempotent.

How provenance survives the round trip

  1. Katz embeds the manuscript commit, spotter name and scope, section ID, byte range, instructions, and manuscript text in each EDSL scenario.
  2. ep run retains that scenario and adds its model and execution metadata to the Results object.
  3. katz spotter ingest verifies the commit and quotation before creating an anchored issue.
  4. The issue stores the spotter, model, Results path, and a stable result hash; the hash prevents duplicate imports.
  5. The complete results.ep remains the durable record of positive and null results for the run.

Model output is still triage, not a referee report. Read every imported draft against the paper and linked materials before confirming it.

09 Investigate, merge, and suggest

Not every first-pass concern is ready for the final report. Record investigation notes, inspect related issues, merge duplicates, and attach a proposed fix only when it is supported.

katz issue list --state open

--state open selects active findings that have moved beyond initial draft triage.

katz issue show ISSUE_ID

Read the complete issue and resolved source context before recording a judgment.

katz issue investigate \
  --id ISSUE_ID \
  --verdict confirmed \
  --notes "Appendix Figure A3 shows pre-period estimates but the text never interprets their joint uncertainty."

--id selects the issue, --verdict records confirmed, rejected, or uncertain, and --notes stores the reasoning.

katz issue suggest \
  --id ISSUE_ID \
  --text "Move the diagnostic into the main text and report a joint pre-trend test."

--id selects the issue and --text appends an actionable proposed response.

katz issue merge \
  --ids ISSUE_ID_1,ISSUE_ID_2 \
  --title "Pre-trend evidence is incomplete and overinterpreted"

--ids supplies a comma-separated set of duplicates to consolidate; --title names the merged issue.

StateUse it when
draftThe concern is plausible but still needs checking.
confirmedThe evidence supports including the finding in the review.
rejectedInvestigation showed that the concern was mistaken or already handled.
resolvedA later revision adequately addressed the finding.
wontfixThe concern is accepted but will not be changed, with a recorded reason.

10 Evaluate review criteria

Issues capture localized findings. Evaluations capture broader judgments such as whether the question is clear, the empirical strategy matches the claims, or limitations are acknowledged honestly.

katz eval init-catalog

Copies the default evaluation criteria into the repository catalog. --preset NAME selects another installed collection.

katz eval catalog --category methods

Lists available criteria; --category methods filters to methods-related judgments.

katz eval enable design_matches_claims

Copies this named criterion into the active version.

katz eval enable limitations_discussed_honestly

Enables a second holistic criterion for the same version.

katz eval respond \
  --name design_matches_claims \
  --text "The design supports a narrower local claim than the abstract makes." \
  --grade B- \
  --suggestion "Align the abstract and conclusion with the estimand."
Argument or optionMeaning
--nameEnabled criterion receiving the response.
--gradeStructured grade from Katz’s accepted scale.
--textRequired evidence-backed rationale.
--suggestionOptional concrete improvement.
katz eval results

Lists completed evaluation responses; --category can filter them.

Do not let a grade replace an argument. The response should state what the paper does, what standard is being applied, and what evidence justifies the assessment.

11 Validate the ledger

Validation checks the internal consistency of registered versions, source hashes, section bounds, issue anchors, and references. Run it before relying on a report and after any manual changes to review state.

katz validate

Checks the active version by default. Add --commit SHA to validate another registered version.

A valid ledger is not necessarily a good review. Validation can prove that an anchor exists and lies within the manuscript. It cannot prove that the interpretation is fair, the evidence is sufficient, or an important issue was noticed.

12 Generate the review report

Once material issues have been investigated and criteria answered, generate a navigable HTML report:

katz report generate --output review.html

--output chooses the destination HTML file. If omitted, Katz uses its default report path.

Open the generated report for this worked example. It opens in a new tab. The relative link works both locally and when the docs/ directory is published from the repository.

The report is a view over the ledger, not a replacement for editorial judgment. Before delivery, read it as a document: consolidate overlapping findings, put claim-changing concerns first, distinguish required changes from optional improvements, and match the tone to the audience.

An internal coauthor review can use direct language, detailed fixes, implementation notes, and unresolved questions. An external referee report should explain its standards, separate major from minor concerns, and avoid prescribing one implementation when alternatives could work.

13 Review a revision

Commit the revised manuscript, register it as a new version, and assess prior findings against the changed text. Never silently move old anchors onto new prose.

git add paper/manuscript.md

Stages the revised manuscript.

git commit -m "Revise manuscript in response to review"

Creates the new Git version that subsequent Katz records will target.

katz paper register \
  --canonical paper/manuscript.md \
  --source-format markdown \
  --source-method revision

--canonical selects the revision, while the format and method record its provenance. Registration uses the new HEAD commit.

katz issue list --state open

Lists findings that still require a revision decision.

katz issue update \
  --id ISSUE_ID \
  --state resolved \
  --reason "Added event-study diagnostics and narrowed the abstract claim."

--id selects a finding, --state resolved closes it for this workflow, and --reason records the evidence for that transition.

katz validate

Checks the revised ledger after all transitions.

katz report generate --output revision-review.html

Writes a separate revision report instead of overwriting the first review.

The old version remains part of the audit trail. This makes it possible to explain what changed, which concerns remain, and why a finding was closed rather than merely disappearing from the report.

15 Review checklist

  1. Commit the repository state that contains the canonical manuscript.
  2. Register the manuscript and verify its checksum and source metadata.
  3. Generate or add section boundaries, then inspect them.
  4. Choose spotters that match the paper, audience, and review scope.
  5. Read the source and file evidence-backed, location-specific issues.
  6. Investigate uncertain findings; merge duplicates and reject false alarms.
  7. Record actionable suggestions without confusing preference with necessity.
  8. Answer holistic evaluation criteria with written reasoning.
  9. Validate the ledger and correct every structural error.
  10. Generate and edit the final report for hierarchy, fairness, and tone.

The completion test. A review is ready when a reader can trace every material conclusion to a specific version and passage, understand why it matters, and see what response would adequately address it.
